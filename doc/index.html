apptronic.net core framework
Introduction
Library is designed to simplify development of, mostly, UI-based applications, but also can be used for backend development.
Framework modules
Context
Context
is a root class for working with library. It manages all other modules and defines some level of responsibility, like screen. App should have at least one root
Context
, which should be global singleton object, and all other
Context
s are sub-contexts of this parent.
Context
s can be organized into tree, each
Context
can have unlimited number of sub-context instances.
Sub-context inherits behavior of the parent
Context
in all areas, except those which is defined in concrete
Context
.
Tree-structure of
Context
instances allows you to organize dependencies in tree structure.
Context guide
Lifecycle
Lifecycle
is defined separately for each
Context
. Generally
Lifecycle
of child
Context
is not bound to parent
Context
, except termination, when
Lifecycle
and
Context
is completely recycled. All other
Lifecycle
management is responsibility of each
Context
by self.
Lifecycle
allows you to define linear structure of
LifecycleStage
s, which can be entered or exited sequentially. Next
LifecycleStage
cannot be entered until previous
LifecycleStage
is not entered. Exiting from previous
LifecycleStage
automatically exits all next
LifecycleStage
s.
Lifecycle
always created with already entered root
LifecycleStage
. Exiting from root
LifecycleStage
is named termination and stops all framework-based activity in
Context
except things, defined to be lifecycle-independent.
Reactive model of framework is bound to
LifecycleStage
s and cannot exist without it. Inside
Context
it uses
Lifecycle
to automatically manage subscriptions, recycle injected instances and perform all other activity. Each subscription, started inside of
LifecycleStage
, automatically unsubscribes when this
LifecycleStage
exits.
Scheduler
Scheduler
responsible for threading. Scheduler defines
Worker
using
WorkerDefinition
. Each
Worker
run's actions according to it threading specification. Each
WorkerDefinition
inside
Scheduler
provides assigned
Worker
by request and defines how worker is provided: separate or same
Worker
for each request.
Each
context
have it's own
Scheduler
but defines no additional behavior by default, inheriting all
WorkerDefinition
s from it's parent. But each
Context
can re-define
Worker
provider for each
WorkerDefinition
or declare another
WorkerDefinition
. In case if
Worker
for requested
WorkerDefinition
is not exists - default
Worker
will be used.
Using
Worker
s allows to define threading behavior by
Context
and make it abstract from platform and from concrete
Component
implementation. Also it provides ability to replace each
Worker
for unit-testing, allowing writing multi-thread tests easily.
Scheduler guide
DependencyProvider
DependencyProvider
is dependency injection implementation. Each
DependencyProvider
defines instances and modules inside of
Context
. Also it can inject them. If instance is not exists in current
Context
- is will search recursively in parent context until core context with no parent. Also each
Context
can redefine items, provided by parent
Context
s. In this case redefined instance will be injected in this
Context
and all child
Context
s.
Component
Component
is an base class for concrete app code implementation. It have access all reactive and threading features, defined by it's
Context
. Each
Component
requires
Context
to work, taking from it
Lifecycle
,
Scheduler
and
DependencyProvider
.
Component
also implements
Context
and can be used as parent for other
Context
.
Context
can be parent for one or many
Component
s, and also each
Component
can contain inner component's to divide complex logic into parts.
All
Components
inside one context shares
Lifecycle
,
Scheduler
and
DependencyProvider
allowing to define all behavior for whole app part, for which concrete
Context
is responsible. Implementing repositories, interactors and all other dependencies, required for domain logic, as
Components
s allows to write simple, testable and reliable code. For unit-testing
Context
can be replaced, for production it can define required behavior and dependencies.
Component guide
ViewModel and ViewModelContext
ViewModel
is implementation of
Components
for development of user interface. It requires to use
ViewModelContext
as it contains corresponding UI
Lifecycle
.
Generally
ViewModel
s are also organized into tree, where root
ViewModel
holds whole application screen. Platform UI controller should manage root
ViewModel
lifecycle. All other
ViewModel
s, which are inside root
ViewModel
and responsible for concrete screens, dialogs or screen parts etc., uses different kinds of
Navigator
s (list/page, stack) which manages it's
ViewModel
s
Lifecycle
according to root
ViewModel
Lifecycle
and concrete
ViewModel
state inside
Navigator
s hierarchy.
Using corresponding adapters allows automatically update UI for concrete platform and write all UI logic inside
ViewModel
s and share it between different platforms.
